# 식사 약속 조율 도메인 명세

## 1. 개요
- 목적: 사용자가 닉네임 기반으로 식사 약속을 생성하고 공유하며, 초대된 참여자들이 가능한 시간대를 제출하도록 하는 반응형 웹 애플리케이션의 핵심 도메인 요구사항을 정의한다.
- 핵심 제약: 로그인 없이 약속 생성과 참여가 가능하며, 확정/잠금/알림 등 부가 상태나 기능을 제공하지 않는다.
- 초기 범위: 단일 시간 슬롯 템플릿(평일 저녁 1회 선택, 주말 점심/저녁 2회 선택)을 사용한다. 추후 템플릿 확장을 고려하여 도메인 모델은 템플릿 다중화를 수용할 수 있어야 한다.

## 2. 주요 도메인 개체
| 개체 | 설명 | 핵심 속성 |
| --- | --- | --- |
| Appointment | 생성자가 정의하는 식사 약속 | `appointment_id`, `title`, `summary`, `time_slot_template_id` |
| Time Slot Template | 특정 날짜 패턴에 맞춰 동적 슬롯을 생성하는 규칙 묶음 | `template_id`, `name`, `description`, `ruleset` |
| Participant Session | 공유 링크로 참여한 개인을 나타내는 닉네임 세션 | `participant_id`, `appointment_id`, `nickname`, `optional_pin`, `responses` |
| Slot Availability | 특정 슬롯에 대한 참여자의 가용성 결과 | `slot_instance_id`, `participant_id`, `is_available`, `submitted_at` |

### 2.1 시간 슬롯 템플릿 규칙 구조
- 규칙은 날짜 패턴과 선택 가능한 식사 시간대 목록으로 구성한다.
- 초기 템플릿 "주간 기본 템플릿" 규칙:
  - 평일(월~금): `dinner` 슬롯 1개만 생성.
  - 주말(토~일): `lunch`, `dinner` 두 슬롯을 생성.
- 슬롯은 실제 데이터베이스에 저장되지 않고, 약속 조회 시 템플릿과 날짜 범위를 기반으로 동적으로 생성한다.
- 추가 템플릿 유형을 확장할 수 있도록 규칙 구조는 다수의 패턴-슬롯 매핑을 지원한다.
- 생성자는 자동 생성된 슬롯을 수동으로 추가하거나 삭제할 수 없다.

## 3. 사용자 흐름
### 3.1 약속 생성
1. 생성자는 제목과 간단한 설명을 입력한다.
2. 생성자는 사용 가능한 시간 슬롯 템플릿 중 하나(초기 구현에서는 주간 기본 템플릿)를 선택한다.
3. 시스템은 `appointment_id`를 기반으로 공유 링크 경로를 생성한다. 별도의 난수 토큰이나 초대 코드는 사용하지 않는다.
4. 약속 정보와 선택된 템플릿 참조만 저장한다.

### 3.2 약속 공유
- 생성자는 약속 식별자 기반으로 생성된 공유 링크를 복사하여 원하는 채널에 전달한다.
- QR 코드, 메시지 공유 등 다른 배포 방식은 제공하지 않는다.

### 3.3 참여 및 응답
1. 참여자는 공유 링크로 접속해 약속을 확인한다.
2. 닉네임(약속별 고유)을 입력해 세션을 생성한다. 추가 개인정보는 수집하지 않는다.
3. 시스템이 템플릿 규칙에 따라 생성한 날짜별 슬롯 목록을 표시한다.
4. 참여자는 가능한 슬롯을 선택한다. 선택하지 않은 슬롯은 자동으로 불가능으로 간주된다.
5. 참여자는 언제든지 동일한 링크와 닉네임(+선택적 PIN)으로 재접속하여 응답을 수정할 수 있다.

## 4. 가시화 및 피드백
- 인터페이스는 슬롯별 가능 인원, 정원(설정 시), 응답률을 요약하여 표시한다.
- 참여자는 자신의 선택과 전체 현황을 동시에 볼 수 있다.
- 확정, 추천, 잠금 등 추가 상태를 제공하지 않는다.

## 5. 데이터 및 보안 제약
- 저장되는 데이터: 약속 기본 정보, 선택된 시간 슬롯 템플릿 참조, 참여자의 닉네임, 슬롯 가용성 응답, 참여자 세션 선택적 PIN.
- 저장하지 않는 항목: 이메일/전화번호 등 개인 식별 정보, 세션 토큰, 자동 생성된 실제 슬롯 인스턴스.
- 데이터 자동 삭제 기능을 제공하지 않는다.
- 보안은 난수화된 링크 수준의 기본 보호만 제공하며, 고급 보안 요구는 범위 밖이다.

## 6. 비기능 요구사항
- 로그인 없는 3단계 이내(약속 생성, 링크 공유, 응답 제출)의 핵심 흐름을 유지한다.
- 반응형 UI를 목표로 하지만 특정 프런트엔드/백엔드 기술 스택은 이 명세 범위에 포함하지 않는다.
- 템플릿과 슬롯 생성 로직은 추후 템플릿 확장을 위해 구성 가능한 구조를 유지한다.
